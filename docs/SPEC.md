# bash-gitaware v2 — Specification

**Status:** draft. This document is the source of truth for what bash-gitaware v2
is and is not. The decisions are recorded as ADRs in `docs/adr/`; the work is
broken into milestones in `docs/PLAN.md`; the render pipeline is drawn in
`docs/diagrams/render-pipeline.md`.

## Problem

A shell prompt runs on every command. It should give you the context you need
(where am I, which branch, is the tree dirty, which runtime/version, did the last
command fail and how long did it take) without clutter, without breaking on
terminals that lack fancy glyphs, and without slowing down the shell. Most bash
prompts show too little (`\u@\h:\w\$`), or break on a non-Nerd-Font terminal
(tofu where glyphs should be, misaligned line wrapping), or are slow (many `git`
subprocesses per prompt). bash-gitaware v1 was a step up but used fragile
box-drawing with manual width arithmetic, and gated glyphs on the *locale* rather
than on whether the *font* actually has them.

## Scope

bash-gitaware v2 is a single-file, zero-dependency bash prompt (bash 4.4+) with:

- A clean two-line prompt: a context line, then the prompt symbol; an optional
  middle line for the last commit subject.
- Modules shown only when relevant: repo-relative path, git branch + status,
  runtime/version (Node, Python, Rust, Go), command duration, exit code.
- Tiered glyphs (nerd > unicode > ascii) with auto-detection that never assumes a
  Nerd Font.
- Modern terminal integration: OSC 133 semantic prompt marks, OSC 7 cwd reporting.
- Transient prompt: the previous prompt collapses to a minimal form after Enter,
  keeping scrollback clean.
- Async rendering: the prompt appears immediately; slow parts (git status on a
  huge repo, runtime version commands on a cold cache) are computed in the
  background and the prompt refreshes when they are ready.
- Presets (`minimal`, `default`, `powerline`, `full`) and ~10 documented
  `BASHGITAWARE_*` knobs.
- A build pipeline: the canonical source is `lib/*.bash`; `new.bashrc` is
  generated by `bin/build.sh` and committed; CI verifies it is current.
- Tests (scenario rendering + a performance budget) and CI (shellcheck + tests
  across bash versions and OSes).
- A `vhs`-generated demo and an `install.sh`.

## Non-goals

- Cross-shell support (zsh, fish). bash-gitaware is bash-native by design. See
  ADR-0006.
- bash 3.2 (stock macOS). Targets bash 4.4+; the README points macOS users to a
  Homebrew bash.
- A configuration wizard or a TOML config file. Configuration is via
  `BASHGITAWARE_*` environment variables and presets — the bash-idiomatic way.
- Replacing the whole `.bashrc`. The shipped `new.bashrc` is a complete, sensible
  `.bashrc` (history, ls colors, aliases, completion, PATH), but the prompt is the
  product; `install.sh` backs up the user's existing file first.

## Users

A developer who lives in the terminal, wants a prompt that is informative but
clean, works on whatever terminal/font is at hand (servers, containers, a fresh
box), and is fast. Secondary: anyone evaluating the implementation — the repo is
small enough to read end to end in about ten minutes, so the code, the ADRs and
the tests are part of what ships.

## User stories

- As a developer, I want the prompt to show the repo name and my path within the
  repo, so I know where I am without parsing a long absolute path.
- As a developer, I want git status (branch, ahead/behind, dirty, stash,
  rebase/merge state) at a glance, so I don't run `git status` constantly.
- As a developer, I want the project's runtime version (Node/Python/Rust/Go)
  shown only when I am in such a project, so I know which toolchain is active
  without clutter everywhere else.
- As a developer, I want the prompt not to break (no tofu, no misaligned boxes, no
  broken line wrapping) on a terminal without a Nerd Font, so I can use the same
  config everywhere.
- As a developer on a huge monorepo, I want the prompt to appear instantly and
  fill in the slow git info a moment later, so the shell never feels laggy.
- As a developer, I want old prompts in my scrollback to collapse to a minimal
  form, so the scrollback stays readable.
- As a developer on a modern terminal (WezTerm, Kitty, VS Code, iTerm2, Ghostty,
  Konsole, Windows Terminal, Warp, ...), I want the prompt to emit OSC 133/7 so
  "jump to previous prompt", "command finished" decorations, and "new tab in same
  directory" work.
- As someone reading the project, I want a spec, ADRs, tests, CI, and a clean git
  history with conventional commits and tagged releases, so I can judge the
  engineering, not just the output.

## Functional requirements

1. **Prompt layout.** Two lines by default. Line 1:
   `[chroot] [container] [user@host ] {path}[ on {branch}[@{hash}]][ {state}][ {ahead}][ {behind}][ {dirty}][ {stash}][ via {runtime}...][ took {duration}][ {exit}]`.
   Line 2 (optional): `↳ {last commit subject}`.
   Line 3: `{symbol} ` — `❯` (unicode/nerd), `>` (ascii), `#` (root); bold green
   on success, bold red on failure. Every ANSI escape is wrapped in `\[ \]`.
2. **Path.** Inside a git work tree: `{repo-name}{path-within-repo}`. Otherwise:
   `~`-relative if under `$HOME`, else absolute. Trim to the last
   `BASHGITAWARE_PATH_MAXDEPTH` components (default 3); the dropped prefix becomes
   an ellipsis (`…` / `...`); the repo name (or `~`) stays visible.
3. **Git.** Branch name; short hash; ahead/behind vs upstream; dirty bit (any
   tracked change or untracked file); stash count; in-progress state
   (`REBASE n/m`, `AM n/m`, `MERGING`, `CHERRY-PICK`, `REVERTING`, `BISECTING`);
   detached HEAD shown as `detached@{hash}`. At most 4 `git` subprocesses per
   prompt (`rev-parse --git-dir --show-toplevel`; `status --porcelain=v2 --branch`,
   reading stops at the first changed-path line; `log -1 --pretty=%s` only if there
   is a commit; `stash list` only if a stash ref file exists). Zero `git`
   subprocesses outside a repo.
4. **Runtime modules.** Node (`package.json` present), Python (a virtualenv/conda
   env active, or `pyproject.toml`/`setup.py`/`setup.cfg`/`requirements.txt`/`Pipfile`/`.python-version`
   present), Rust (`Cargo.toml`), Go (`go.mod`). Version from a version file when
   present (no subprocess), else from `node --version` / `python --version` /
   `rustc --version` / `go env GOVERSION`. Result cached per
   `(PWD, VIRTUAL_ENV, CONDA_DEFAULT_ENV)`. The Python module includes the
   environment name (the parent directory's name when the venv directory is
   generically named `.venv`/`venv`/`env`/`.env`/`virtualenv`). Disable with
   `BASHGITAWARE_RUNTIME=0`.
5. **Glyph tiers.** `nerd` (the Powerline branch glyph plus the unicode set),
   `unicode` (arrows, a dot, etc. — only glyphs present in virtually every
   monospace font), `ascii` (`^ v * s` plus word fallbacks such as `commit:` and
   `exit N`). Auto: `unicode` if the locale is UTF-8, else `ascii`; `nerd` is only
   ever selected explicitly (`BASHGITAWARE_GLYPHS=nerd` or
   `BASHGITAWARE_NERD_FONT=1`) — a UTF-8 locale does not imply the font has Nerd
   glyphs. Force any tier with `BASHGITAWARE_GLYPHS`.
6. **Color.** A 256-color palette: steel-blue path, purple branch, yellow
   status/duration, green/red prompt symbol, grey connectors and short hash,
   per-runtime colors. Disabled entirely if `NO_COLOR` is set or the terminal has
   no color capability.
7. **OSC integration.** Emit OSC 133 `A` (prompt start) and `B` (prompt end)
   around the prompt; `C` (command start) from the preexec hook; `D;{exit}`
   (command end) at the start of the next prompt cycle. Emit OSC 7
   (`file://{host}{pwd}`) each prompt. Emit OSC 0/2 (window title) when the
   terminal supports it. Disable OSC 133/7 with `BASHGITAWARE_OSC=0`.
8. **Transient prompt.** When a command is submitted, redraw the just-shown prompt
   as a minimal one-line form (`{symbol} ` only), so the scrollback shows minimal
   prompts and only the live prompt is full. Handle multi-line prompts, the first
   prompt, empty submissions, `Ctrl-C` at the prompt, and `clear`. Disable with
   `BASHGITAWARE_TRANSIENT=0`.
9. **Async rendering.** On each prompt: render immediately with the cheap info
   (path, env detection, `git rev-parse`, cached runtime). If the expensive info
   (full `git status`, cold-cache runtime versions) is not ready, show a subtle
   placeholder and compute it in a background job; when it completes, refresh the
   prompt in place — or, as a documented fallback, on the next prompt. Never leave
   a stale background job; never corrupt the terminal. Disable with
   `BASHGITAWARE_ASYNC=0` (the prompt is then fully synchronous).
10. **Presets.** `BASHGITAWARE_PRESET`: `minimal` (one line, no commit line, no
    runtime, ascii-leaning), `default` (everything above), `powerline` (colored
    arrow segments; needs a Powerline/Nerd font), `full` (default plus more
    verbose git status plus always show user@host). A preset sets defaults that
    individual `BASHGITAWARE_*` variables still override.
11. **Knobs.** `BASHGITAWARE_GLYPHS`, `BASHGITAWARE_NERD_FONT`,
    `BASHGITAWARE_PRESET`, `BASHGITAWARE_COMMIT_LINE`, `BASHGITAWARE_RUNTIME`,
    `BASHGITAWARE_OSC`, `BASHGITAWARE_TRANSIENT`, `BASHGITAWARE_ASYNC`,
    `BASHGITAWARE_TIMER_THRESHOLD`, `BASHGITAWARE_PATH_MAXDEPTH`,
    `BASHGITAWARE_SHOW_HOST`, plus the standard `NO_COLOR`. All documented in the
    file header and the README.
12. **Build.** `bin/build.sh` concatenates `lib/[0-9][0-9]-*.bash` (in numeric
    order) with the file header into `new.bashrc`. `make build`. `new.bashrc` is
    committed; CI fails if it differs from a fresh build.
13. **Install.** `install.sh`: detect the bash version (warn if < 4.4); back up
    `~/.bashrc` to `~/.bashrc.bak-{timestamp}`; copy `new.bashrc` to `~/.bashrc`;
    optionally append a chosen preset/glyph line; print next steps.

## Non-functional requirements

- **Performance.** The synchronous render path (async disabled, warm caches, a
  small repo) completes well under a person-perceptible delay; a CI test asserts a
  budget (target on the CI runner: under ~30 ms). The async path makes the
  *perceived* time independent of repo size. At most 4 `git` subprocesses per
  prompt in a repo; 0 in a non-repo; runtime version commands run at most once per
  directory.
- **Robustness.** No tofu on a non-Nerd-Font terminal at the default tier. No
  misaligned output (no manual box-width arithmetic). No broken line wrapping
  (every ANSI escape inside `\[ \]`). Survives a terminal resize and `Ctrl-C` at
  the prompt. Degrades gracefully when `git`, `tput`, `dircolors` or a runtime
  binary is missing.
- **Portability.** bash 4.4, 5.0, 5.1, 5.2 and current bash, on Linux and macOS
  (Homebrew bash).
- **Security.** Minimal attack surface. No `eval` of untrusted input; branch
  names, commit subjects and paths are inserted as data, never re-evaluated. The
  (low) residual risk and its mitigation are documented in `SECURITY.md`.
- **Readability.** The whole repo is readable end to end in roughly ten minutes:
  modular `lib/`, an ADR per significant decision, a render-pipeline diagram,
  conventional commits, tagged releases.

## Success criteria

- All functional requirements implemented; all `BASHGITAWARE_*` knobs work; all
  presets render.
- CI green on the full matrix: shellcheck clean; `new.bashrc` matches a fresh
  build; scenario tests pass; the performance budget is met.
- A reader can: open `SPEC.md` and understand the intent; read `docs/adr/` and
  understand the decisions; read `lib/` and understand the implementation; read
  the git log and see a clean, conventional history with tagged releases; open the
  README and see the demo and a clear install path; run `make test` and get green.
- `v2.0.0` tagged with a GitHub Release.
- The "broken symbols" failure mode of v1 is gone (verified: the default tier on a
  non-Nerd-Font terminal, plus the README troubleshooting section).

## Out of scope for v2 (possible later)

- A right-side prompt (RPROMPT-style).
- A vi-mode indicator.
- More runtime modules (Ruby, Java, .NET, Deno, Bun, Zig, ...).
- Per-segment custom formats / a templating mini-language.
- OSC 9;4 (taskbar progress) for long-running commands.
